# 골프왕부킹 MVP PRD
_프로젝트 폴더명: `golfking-booking`_

## 1. 제품 개요

### 1.1 제품명
**골프왕부킹**

### 1.2 한 줄 소개
사용자가 **희망 라운드 날짜**를 입력하면, 해당 날짜에 맞춰 **각 골프장의 예약 오픈일/오픈시간**을 한눈에 확인할 수 있는 **골프 예약 오픈 정보 탐색 서비스**.

### 1.3 제품 정의
골프왕부킹은 사용자를 대신해 예약을 수행하는 자동화 서비스가 아니라,  
**“언제 어떤 골프장이 예약을 여는지”를 구조화해서 보여주는 정보형 서비스**다.

핵심 가치는 다음과 같다.

- 사용자는 원하는 플레이 날짜를 기준으로
- 어떤 골프장이
- 언제 예약이 열리는지
- 어느 지역에 있는지
- 어디서 예약해야 하는지(홈페이지/예약 페이지)
를 빠르게 파악할 수 있다.

즉, 골프왕부킹은 **예약 실행 도구**가 아니라 **예약 타이밍 인텔리전스 도구**다.

### 1.4 핵심 가치 제안
- 골프 예약의 가장 큰 불편인 **“오픈 타이밍 파악의 어려움”**을 해결
- 수작업으로 골프장별 정책을 찾고 메모하던 과정을 단순화
- 날짜 중심, 지역 중심 탐색으로 실제 예약 준비 시간을 줄임
- 링크 연결을 통해 사용자가 바로 공식 예약 채널로 이동 가능

## 2. 문제 정의

### 2.1 사용자 문제
국내 골프장 예약은 보통 아래와 같은 특성을 가진다.

- 골프장마다 **예약 오픈 정책**이 다르다
  - 예: 3주 전 오전 9시 오픈
  - 예: 매주 월요일 다음 주차 오픈
  - 예: 매월 특정일 일괄 오픈
- 공식 홈페이지, 외부 예약 플랫폼, 전화 예약 등 **채널이 제각각**이다
- 사용자는 원하는 라운드 날짜가 생기면, 여러 골프장을 돌아다니며 직접 정책을 확인해야 한다
- 정책이 텍스트로 흩어져 있어 **“내가 가고 싶은 날짜 기준으로 역산해서”** 보기 어렵다
- 지역, 거리, 선호 골프장 기준으로 빠르게 후보를 정리하기 어렵다

### 2.2 현재 대안의 한계
현재 사용자는 보통 아래 방식으로 해결한다.

- 네이버 검색 / 골프장 홈페이지 개별 확인
- 카카오톡 메모 / 엑셀 / 개인 캘린더에 수기 정리
- 골프 커뮤니티/카페에서 정보 탐색
- 자주 가는 골프장만 기억에 의존

이 방식의 문제점:

- 시간 소모가 크다
- 정보 최신성이 떨어진다
- 실수로 오픈 시간을 놓치기 쉽다
- 신규 골프장 탐색이 어렵다
- 예약 성공 확률을 높이기 위한 체계적인 판단이 어렵다

### 2.3 해결해야 할 핵심 문제
골프왕부킹의 MVP는 다음 문제를 해결해야 한다.

> “내가 치고 싶은 날짜에 대해, 어떤 골프장이 언제 예약을 오픈하는지 빠르게 알 수 있어야 한다.”

## 3. 타깃 사용자

### 3.1 1차 타깃
**직접 예약을 챙기는 개인 골퍼**

특징:
- 주말/공휴일 라운드를 자주 계획함
- 인기 시간대 예약 경쟁을 자주 경험함
- 3~10개 정도의 관심 골프장을 관리하고 싶어함
- 골프장 예약 규칙을 일일이 확인하는 데 피로를 느낌

### 3.2 2차 타깃
**골프 일정 조율 담당자**

예:
- 3~4인 팀의 총무 역할을 하는 사용자
- 지인/고객/동호회 라운드를 잡는 사람
- 법인/소규모 모임 일정 조율자

니즈:
- 특정 날짜 기준으로 가능한 후보를 빠르게 보고 싶음
- 지역별로 골프장을 추려 비교하고 싶음
- 링크 클릭 후 바로 예약 채널로 넘어가고 싶음

### 3.3 제외 대상
초기 MVP에서는 아래 사용자는 우선순위가 낮다.

- 예약 자동화/대리 예약을 원하는 사용자
- 티타임 가격 비교가 핵심인 사용자
- 실시간 잔여 티타임 재고 확인이 필요한 사용자
- 해외 골프장 예약 사용자

## 4. 목표 / 비목표

### 4.1 제품 목표

#### 비즈니스 목표
- 골프 예약 정보 탐색 문제를 명확히 해결하는 MVP를 빠르게 출시
- 반복 방문 가능한 정보형 서비스의 기반 확보
- 향후 알림/개인화/프리미엄 기능으로 확장 가능한 데이터 구조 확보

#### 사용자 목표
- 원하는 라운드 날짜를 입력하면 관련 예약 오픈 일정을 즉시 확인
- 지역별로 후보 골프장 추리기
- 공식 예약 채널로 바로 이동
- “예약 타이밍 놓침” 가능성을 줄이기

#### 제품 목표 KPI (MVP 가설)
- 사용자가 첫 방문 후 **1분 이내에 의미 있는 결과를 확인**
- 검색 사용자 중 **링크 클릭률(예약/홈페이지 이동)** 확보
- 재방문율로 정보 서비스의 반복 효용 검증
- 관리자 입장에서 최소 인력으로 데이터 운영 가능

### 4.2 비목표 (Non-goals)
MVP 단계에서 하지 않는 것:

- 자동 로그인 / 자동 예약 / 매크로 / 봇 기능
- 실시간 티타임 재고 수집 및 판매
- 결제 기능
- 대규모 커뮤니티 기능
- 사용자 간 후기/평점 시스템
- 가격 비교 엔진
- AI 추천 엔진
- 전국 모든 골프장 완전 커버리지 보장

## 5. MVP 범위

### 5.1 포함 범위
MVP는 다음 기능을 포함한다.

1. **희망 플레이 날짜 입력**
2. **해당 날짜 기준 예약 오픈 일정 계산/표시**
3. **골프장 목록/캘린더 형태 조회**
4. **지역 필터링**
5. **골프장 상세 정보 표시**
   - 골프장명
   - 지역
   - 예약 오픈 정책 요약
   - 예약 오픈 예정 일시
   - 지도 링크
   - 홈페이지 링크
   - 예약 페이지 링크
6. **관리자 데이터 등록/수정 기능**
7. **정책/링크 최신성 관리용 운영 화면**
8. **기본 반응형 웹 지원**

### 5.2 제외 범위
- 회원 간 소셜 기능
- 모바일 앱
- 푸시 알림/문자 알림
- 개인 맞춤 watchlist
- 실시간 외부 사이트 상태 모니터링
- 자동 크롤링 기반 전면 운영
- 예약 성공률 분석 대시보드

### 5.3 MVP 성공 기준
다음 조건을 만족하면 MVP로서 의미가 있다.

- 핵심 골프장 데이터셋(예: 50~150개)을 운영 가능
- 사용자가 날짜 입력 후 오픈 일정을 직관적으로 이해 가능
- 관리자가 비개발자 수준으로 데이터 유지보수 가능
- 법적/운영 리스크가 과도하지 않음

## 6. 사용자 스토리

### 6.1 일반 사용자 스토리
1. 사용자는 4월 20일에 라운드하고 싶다.
2. 서비스에 날짜를 입력한다.
3. 해당 날짜에 대해 각 골프장의 예약 오픈 예정 일시를 본다.
4. 경기 남부 지역만 필터링한다.
5. 오픈 시간이 빠른 순서로 목록을 확인한다.
6. 예약 페이지 링크를 눌러 공식 채널로 이동한다.

### 6.2 총무형 사용자 스토리
1. 사용자는 주말 4인 라운드를 준비 중이다.
2. 특정 날짜에 예약 가능한 골프장 후보를 지역별로 좁힌다.
3. 각 골프장의 오픈 정책과 링크를 비교한다.
4. 우선순위를 정한 뒤 오픈 시간에 맞춰 예약 시도 계획을 세운다.

### 6.3 관리자 스토리
1. 운영자는 신규 골프장을 등록한다.
2. 예약 정책(예: 라운드 21일 전 오전 9시)을 설정한다.
3. 지도/홈페이지/예약 링크를 등록한다.
4. 정책이 변경되면 수정한다.
5. 잘못된 링크나 오래된 정책을 점검한다.

## 7. 기능 요구사항

### 7.1 사용자 기능 요구사항

#### FR-1. 날짜 입력
- 사용자는 라운드 희망 날짜를 입력할 수 있어야 한다.
- 입력 방식은 달력(Date Picker)을 기본으로 한다.
- 과거 날짜는 기본적으로 선택 불가 처리한다.
- 기본 진입 시 “오늘 기준 향후 일정” 중심 UX를 제공한다.

#### FR-2. 예약 오픈 일정 계산
- 시스템은 사용자가 선택한 플레이 날짜를 기준으로 각 골프장의 예약 오픈 일시를 계산해야 한다.
- 예약 정책은 골프장별로 다르므로, 정책 타입에 따라 계산 로직이 달라져야 한다.
- 최소한 아래 정책 타입을 지원해야 한다.
  - **N일 전 오픈**
  - **특정 요일 기준 오픈**
  - **월 단위/특정일 일괄 오픈**
  - **수동 지정형**
- 계산 불가한 경우 “운영 확인 필요” 상태를 표시할 수 있어야 한다.

#### FR-3. 결과 목록 표시
- 결과는 리스트 형태로 표시되어야 한다.
- 각 결과 항목에는 최소 아래 정보가 포함되어야 한다.
  - 골프장명
  - 지역
  - 예약 오픈 예정 일시
  - 정책 요약 문구
  - 공식 홈페이지 링크
  - 예약 링크
  - 지도 링크
- 정렬 기준:
  - 기본: 예약 오픈 시점 오름차순
  - 옵션: 지역명, 골프장명

#### FR-4. 캘린더/타임라인 뷰
- MVP에서 완전한 복잡한 캘린더가 아니어도 되지만,
- 사용자에게 “언제 오픈되는지”를 직관적으로 보여주는 시각화가 필요하다.
- 최소 구현 대안:
  - 날짜별 그룹화 리스트
  - 오픈 일정 캘린더 뷰
  - 시간순 타임라인
- MVP에서는 구현 난이도를 고려해 **리스트 + 날짜 그룹 표시**를 기본안으로 한다.
- 캘린더 뷰는 있으면 좋지만 2차 우선순위로 둘 수 있다.

#### FR-5. 지역 필터
- 사용자는 지역 기준으로 골프장을 필터링할 수 있어야 한다.
- 최소 지역 단위:
  - 시/도(예: 경기, 강원, 충북, 경북 등)
- 추후 확장을 고려해 하위 지역(예: 경기 남부/북부, 시군구)을 저장 가능해야 한다.

#### FR-6. 골프장 상세 정보 확인
- 사용자는 특정 골프장의 상세 정보를 볼 수 있어야 한다.
- 상세 정보에는 다음이 포함된다.
  - 골프장명
  - 주소/지역
  - 설명(선택)
  - 예약 정책 원문/요약
  - 최근 정책 확인일
  - 공식 사이트 링크
  - 예약 링크
  - 지도 링크
  - 메모(운영자 작성 요약)

#### FR-7. 링크 이동
- 사용자는 홈페이지, 예약 페이지, 지도 링크로 바로 이동할 수 있어야 한다.
- 외부 링크는 새 탭으로 여는 것을 기본으로 한다.
- 링크가 없는 경우 명확히 표시한다.

#### FR-8. 검색/필터 조합
- 사용자는 날짜 + 지역 필터 조합으로 결과를 좁힐 수 있어야 한다.
- 추후 확장 가능한 필드:
  - 회원제/대중제
  - 산악형/평지형
  - 선호 골프장
- 단, MVP에서는 날짜 + 지역만 필수 지원한다.

#### FR-9. 결과 상태 표시
각 골프장 결과는 다음과 같은 상태를 가질 수 있어야 한다.
- 예약 오픈 예정
- 오늘 오픈
- 이미 오픈됨
- 계산 불가
- 정보 확인 필요

#### FR-10. 기본 SEO/공유 메타
- 랜딩 페이지 및 주요 결과 페이지는 기본 SEO 메타를 가져야 한다.
- 공유 시 서비스 취지가 명확히 보이도록 title/description 설정 필요

### 7.2 관리자 기능 요구사항

#### FR-11. 관리자 인증
- 관리자만 골프장 데이터 등록/수정 가능해야 한다.
- 초기 MVP에서는 Supabase Auth 기반 이메일 로그인 또는 단일 관리자 계정으로 충분하다.

#### FR-12. 골프장 등록/수정
- 운영자는 골프장을 생성/수정/비활성화할 수 있어야 한다.
- 필수 입력 항목:
  - 골프장명
  - 지역
  - 예약 정책 타입
  - 정책 파라미터
  - 홈페이지 링크 또는 예약 링크 중 최소 1개
- 선택 입력:
  - 주소
  - 지도 링크
  - 운영 메모
  - 정책 확인일

#### FR-13. 예약 정책 관리
- 운영자는 예약 정책을 구조화된 필드로 저장해야 한다.
- 자유 텍스트만 저장하는 방식은 계산 자동화에 불리하므로 불가하다.
- 단, 예외 대응을 위해 정책 원문 텍스트도 보조 저장 가능해야 한다.

#### FR-14. 데이터 검수 상태 관리
각 골프장 데이터는 아래 상태를 가질 수 있어야 한다.
- 초안
- 검수완료
- 업데이트 필요
- 비노출

#### FR-15. 운영 변경 이력
- 누가, 언제, 어떤 필드를 수정했는지 최소 수준의 변경 이력이 있어야 한다.
- MVP에서는 full audit log까지는 아니어도 `updated_at`, `updated_by` 정도는 필수다.

### 8.2 정책 데이터 구조 요구사항
골프장 정책은 최소 아래 형태를 지원해야 한다.

#### 타입 A: N일 전 오픈
예:
- 플레이일 21일 전 오전 9시 오픈

필드:
- policy_type = `days_before`
- days_before_open = 21
- open_time = `09:00`

#### 타입 B: 특정 요일 기준 오픈
예:
- 매주 월요일 오전 10시에 다음 주 예약 오픈

필드:
- policy_type = `weekday_rule`
- open_weekday = `MON`
- open_time = `10:00`
- rule_interpretation = “다음 주차”

#### 타입 C: 월별 일괄 오픈
예:
- 매월 1일 오전 9시에 익월 전체 오픈

필드:
- policy_type = `monthly_batch`
- monthly_open_rule = JSON 또는 구조화 필드

#### 타입 D: 수동 지정형
예:
- 정책이 자주 바뀌거나 비정형인 경우
- 운영자가 수동 메모와 예정 시점을 입력

필드:
- policy_type = `manual`
- manual_rule_text
- optional_manual_open_datetime

### 8.3 데이터 품질 요구사항
- 각 골프장 정책은 **최근 확인일**이 있어야 한다.
- 링크는 저장 시 URL 유효성 검사를 한다.
- 정책 계산 실패 항목은 운영 검수 큐에 올라갈 수 있어야 한다.
- 동일 골프장 중복 등록 방지 로직 필요

### 8.4 초기 데이터 확보 기준
MVP 초기에는 전국 전체보다 **핵심 지역/인기 골프장 중심**으로 시작한다.

권장 초기 범위:
- 수도권/강원/충청 일부 중심
- 50~150개 골프장
- 정책이 비교적 명확한 골프장 우선

## 9. 관리자 / 운영 요구사항

### 9.1 운영 원칙
MVP 운영은 자동 수집보다 **정확도 중심의 반수동 운영**이 적합하다.

이유:
- 골프장 정책은 예외가 많고 자주 바뀔 수 있음
- 사이트 구조가 제각각이라 자동화 비용이 높음
- 초기에는 제품 적합성 검증이 우선

### 9.2 운영자 업무 플로우
1. 신규 골프장 후보 수집
2. 공식 홈페이지/예약 페이지에서 정책 확인
3. 정책 구조화 입력
4. 검수 완료 후 노출
5. 정책 변경 여부 주기 점검
6. 사용자 피드백 또는 오류 제보 반영

### 9.3 운영 화면 요구사항
관리자 페이지에서 최소 아래 기능 필요:
- 골프장 목록 조회
- 검색/필터(지역, 상태)
- 신규 등록
- 수정
- 비노출 처리
- 최근 확인일 확인
- 업데이트 필요 항목 보기

### 9.4 운영 알림/점검
MVP 단계에서는 복잡한 자동 알림 대신 아래 수준이면 충분하다.
- 최근 확인일이 오래된 항목 표시
- 링크 누락 항목 표시
- 계산 실패 항목 표시
- 비정형 정책 항목 표시

### 9.5 사용자 피드백 수집
- “정보가 다릅니다” 신고 버튼 또는 문의 링크 제공 고려
- MVP에서는 간단한 폼/이메일 연동으로 충분

## 10. 기술 아키텍처 가정

### 10.1 기술 스택
- **Frontend / Fullstack:** Next.js
- **Hosting / Deployment:** Vercel
- **Database / Auth:** Supabase
- **Admin Auth:** Supabase Auth
- **Storage:** Supabase Storage (필요 시)
- **Analytics:** Vercel Analytics / GA4 / PostHog 중 선택
- **Maps:** 외부 지도 링크 우선(네이버지도/카카오맵/구글맵), 지도 임베드 API는 후순위

### 10.2 아키텍처 방향
#### 사용자 화면
- Next.js App Router 기반 웹 앱
- 서버 컴포넌트 + 필요한 부분만 클라이언트 컴포넌트 사용
- 검색 결과는 SSR/ISR 또는 서버 액션 기반으로 제공 가능

#### 백엔드
- Supabase Postgres에 골프장/정책 데이터 저장
- 예약 오픈 계산 로직은 Next.js 서버 측 함수 또는 API route에서 수행
- 반복 계산 비용이 크지 않으므로 MVP에서는 **실시간 계산** 우선
- 필요 시 계산 결과 캐시 도입

#### 관리자 기능
- `/admin` 경로 별도 구성
- Supabase Auth로 접근 제어
- Row Level Security 또는 애플리케이션 레벨 권한 제어 적용

### 10.3 계산 로직 설계 원칙
- 정책 계산 함수는 타입별로 모듈화
- 예외 규칙은 수동 정책으로 우회 가능
- 계산 결과에는 근거 문구(policy summary)를 함께 반환
- 정책 로직 변경 시 버전 관리 가능하도록 설계

### 10.4 성능 가정
- 초기 트래픽은 높지 않으므로 과도한 분산 아키텍처 불필요
- 사용자 입력 1회당 50~150개 골프장 계산은 충분히 처리 가능
- 응답 시간 목표: 주요 조회 2초 이내

### 10.5 보안 가정
- 관리자 경로는 인증 필수
- 공개 API는 rate limit 고려
- 외부 링크는 `noopener`, `noreferrer` 등 기본 보안 처리
- 관리자 입력 필드는 XSS 방어 필요

## 11. 법률 / 리스크 노트

### 11.1 서비스 성격 명확화
서비스는 다음 성격을 명확히 해야 한다.

- 공식 예약 대행 서비스가 아님
- 예약 보장 서비스가 아님
- 정보 제공 서비스임
- 실제 예약 정책 및 가능 여부는 골프장 공식 채널 기준임

### 11.2 필수 고지 문구 방향
다음 성격의 고지가 필요하다.
- 예약 오픈 정책은 변경될 수 있음
- 최신 정보는 공식 홈페이지/예약 채널에서 최종 확인 필요
- 본 서비스는 참고용 정보 제공 서비스임
- 외부 사이트의 정확성/가용성은 보장하지 않음

### 11.3 크롤링/데이터 수집 리스크
- 골프장 사이트의 이용약관/robots 정책 확인 필요
- MVP에서는 무리한 자동 크롤링보다 수동 검수 중심 운영 권장
- 외부 사이트 내용을 대량 복제/재배포하는 구조는 지양
- 정책 요약/링크 연결 중심으로 운영

### 11.4 상표/브랜드 리스크
- 골프장 이름 표기는 사실 기반 정보 제공 범위 내 사용
- 각 골프장 상호/상표 사용 시 오인 유발 금지
- 제휴/공식 파트너처럼 보이는 표현 금지

### 11.5 개인정보 리스크
MVP에서 회원 기능이 약하면 개인정보 리스크는 낮다.
다만 아래는 필요:
- 문의 폼 운영 시 개인정보 처리방침
- 관리자 계정 보안 관리
- 로그 저장 최소화
